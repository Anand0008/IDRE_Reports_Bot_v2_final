Join Graph Utility

Automatically resolves JOIN paths between any two tables using the database's foreign-key graph — so the SQL writer always knows how to connect tables, not just which ones to use.

Why it exists

⚠️

The Problem (before 1.3)

The schema mapper returned a list of table names. The SQL writer knew what tables were relevant but had no information about how to connect them.

The LLM had to guess JOIN conditions from column names alone — leading to wrong joins, missing intermediate tables, and Cartesian products.

-- LLM guessed (often wrong)
SELECT c.id, p.amount
FROM `case` c
JOIN `payment` p
  ON c.id = p.caseId
-- ❌ payment has no caseId column!
✅

The Solution (after 1.3)

The join graph traverses the FK relationships extracted from the database and injects exact, correct JOIN clauses directly into the schema context.

The LLM receives ready-to-use SQL snippets — it no longer needs to guess column names or intermediate tables.

-- Join graph resolved (correct)
SELECT c.id, p.amount
FROM `case` c
JOIN `case_payment_allocation` cpa
  ON c.id = cpa.caseId
JOIN `payment` p
  ON cpa.paymentId = p.id
-- ✅ Correct 2-hop path via allocation

payment

Path > 2 hops — skipped (context stays clean)

Paths longer than 2 hops are deliberately excluded. Injecting long speculative chains would bloat the prompt and introduce noise — the LLM handles these rare cases with the table + FK column names already present in the schema context.

How the algorithm works

1
Build bidirectional FK graph (once, on first import)
Reads schema_catalog.json. For every FK entry {column → references_table.references_column}, adds two edges: forward (A→B) and reverse (B→A). Stored as a singleton dict in memory.
2
Schema Mapper returns top-K relevant tables
ChromaDB vector search finds the 6 most semantically relevant tables for the user's query. This list is passed to the join graph.
3
All-pairs BFS on the matched tables (≤2 hops)
For every pair (T1, T2) in the matched table list, BFS finds the shortest FK path. Paths with more than 2 hops are discarded to avoid noise.
4
Emit JOIN clause strings
Each edge in the path becomes a JOIN `table` ON `a`.`col` = `b`.`col` string. Pairs are labelled as "direct FK" or "via `intermediate_table`".
5
Append to schema_context in GraphState
The join hint block is appended to the schema context string under the header "Suggested JOIN paths between relevant tables" and flows to the SQL Writer via GraphState.

Public API — utils/join_graph.py

get_join_path(start: str, end: str) → list[tuple] | None
startSource table name (must exist in schema_catalog.json)
endTarget table name
Returns list[(from_table, from_col, to_table, to_col)] — shortest BFS path of FK edges. Returns None if no path exists, [] if start == end.
get_join_context(table_names: list[str]) → str
table_namesList of tables returned by the schema mapper (top-K from ChromaDB)
Returns a formatted string block with all ≤2-hop JOIN conditions between every pair. Returns "" if no paths found. Intended to be appended directly to schema_context in GraphState.
get_graph_stats() → dict
Returns {"tables": int, "fk_edges": int} — used by the Schema Mapper agent trace entry.
